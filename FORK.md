# Fork This Archive for Your Own ESPN League

A handoff doc for the friend who got this repo from Bennett. Drop this file
into Claude Code, Cursor, or another agentic AI alongside the cloned repo
and tell the AI:

> Read FORK.md and walk me through forking this site for my ESPN league.

This file is written **for the AI**, not for you. The AI is supposed to:

1. Interview you for the five inputs below.
2. Edit the right files with what you give it.
3. Run the pipeline.
4. Deploy.

You just answer questions and approve diffs.

---

## What this repo is

A weekend-project archive site for an ESPN fantasy football league:

- **`pipeline/`** — Python. Pulls all historical season data from ESPN via the
  `espn_api` package and writes canonical JSON into `data/`.
- **`data/`** — Committed JSON dataset (regenerated by pipeline; one file per
  season under `data/seasons/`, plus aggregates like `records.json`,
  `owners.json`, `players.json`, `keepers.json`).
- **`web/`** — Next.js 16 App Router site (TypeScript + Tailwind 4) that
  reads `data/` at build time. Deployed to Vercel from `web/` as the root.
- **`.github/workflows/refresh.yml`** — Weekly cron that re-runs the pipeline
  and commits any diff, triggering a Vercel redeploy.

The site has these pages: Home, Trophies (`/champions`), Playoffs, What If,
Records, Players, Seasons, Owners (`/teams`), Keepers. All work
out-of-the-box once the data files exist.

## AI: do this in order

### Step 1 — Interview the user

Before touching any files, ask the user for **all five** of these. Don't
guess defaults; the answers go into specific lines. Use whatever interactive
question UI your harness has (Claude Code's AskUserQuestion, Cursor's chat,
etc.) and group the questions so the user can answer in one pass.

| # | Question | What it's used for |
|---|----------|--------------------|
| 1 | **ESPN league ID** (integer in the league URL, e.g. `113984`) | `pipeline/config.py: LEAGUE_ID` |
| 2 | **First year** the league existed on ESPN (e.g. `2018`) | `pipeline/config.py: FIRST_YEAR` |
| 3 | **Last completed season** with data (e.g. `2025`) | `pipeline/config.py: CURRENT_YEAR` |
| 4 | **League / site name** for branding (e.g. `"The Big Board"`, plus a 2-letter mark like `BB` for the logo square) | `web/src/app/layout.tsx` (header, footer, metadata), `web/src/app/page.tsx` (hero), every other page's `<title>` and any visible "Steak Frites" string |
| 5 | **ESPN cookies** (`espn_s2` and `SWID`) — required even for public leagues to pull historical seasons. Tell the user how to get them: log in at fantasy.espn.com → DevTools → Application → Cookies → copy `espn_s2` (a long opaque string) and `SWID` (keep the curly braces). Treat these as secrets — never write them into a file. They go into env vars + GitHub Actions secrets only. | Env vars for `python -m pipeline.build`; GitHub Secrets for the weekly refresh workflow |

Then ask about **roster format and keeper rules**, because those have
non-default plumbing:

#### 5a. Roster format

> What does your starting lineup look like — specifically, how many QBs
> and any flex/superflex spots? (e.g. "1QB + 2RB/3WR/1TE/1FLEX/1K/1D",
> "2QB + 2RB/2WR/1TE/1FLEX", "Superflex").

**For a 2QB or Superflex league:** the pipeline doesn't need any code
changes — `pipeline/players.py` already keys "did this player start?" off
of `slot_position not in {BE, IR}`, which is roster-format agnostic. ESPN
reports the slot the player actually filled (`QB`, `OP` for superflex, etc.)
and the math just works.

What **does** change in a 2QB/superflex league:

- The all-time Players page (`/players`) groups single-week performances by
  position. `POSITION_ORDER` in `pipeline/players.py` is `["QB", "RB", "WR",
  "TE", "K", "D/ST"]`. That's fine for 2QB (QBs are still QBs). Leave it.
- The Regular-Season MVP score formula in `pipeline/players.py` weights
  starting points × team win rate. Still correct in 2QB; no change needed.
- If you want a "Top Superflex/OP weeks" leaderboard later, that's an
  additive feature, not a fix.

**If the user says "yeah just regular 1QB," skip this section entirely.**

#### 5b. Keeper rules

Open `pipeline/config.py` and show the user the current `KEEPER_RULES`
block:

```python
KEEPER_RULES = {
    "max_total": 2,
    "blocked_rounds": [1, 2, 3],
    "max_rounds_4_to_7": 1,
    "max_rounds_8_to_16": 2,
    "free_agent_round": 16,
}
```

Translate the keys:

- `max_total` — how many keepers a team gets total per draft.
- `blocked_rounds` — rounds from which players **cannot** be kept (here,
  any player drafted in rounds 1–3 is ineligible).
- `max_rounds_4_to_7` — cap on keepers whose previous-year draft round
  fell in 4–7.
- `max_rounds_8_to_16` — cap on keepers whose round was 8 or deeper.
- `free_agent_round` — round assigned to free-agent keepers (Bennett's
  league treats an FA add as a last-round keeper).

Ask the user:

> What are your league's keeper rules? "No keepers" is a fine answer.

Map their answer to those fields. If their rules use different bands
(e.g. "round 5–9: max 1, round 10+: max 2"), tell the user **the keeper
validator schema is hard-coded to two bands (`rd_4_7`, `rd_8_16`)** and you
will need to edit:

- `pipeline/keepers.py: keeper_round_band()` — change the boundary numbers.
- `pipeline/keepers.py: evaluate_keeper_set()` — change the band counts.
- `web/src/lib/keeper-rules.ts` — the same logic mirrored in TypeScript;
  update the band names and limits so the frontend planner matches.

For "no keepers" leagues, the simplest change is to set `max_total: 0`
and tell the user the Keepers tab will be empty (or hide it from the nav
in `web/src/app/layout.tsx`).

#### 5c. Sleeper / off-platform years (optional)

If the league spent any year on Sleeper, Yahoo, or another platform with no
ESPN data, ask:

> Were there any seasons where you played on a different platform? The
> archive can include the year as a placeholder but exclude it from
> all-time records.

Add each one to `EXCLUDED_RECORD_YEARS` in `pipeline/config.py`:

```python
EXCLUDED_RECORD_YEARS: dict[int, str] = {
    2021: "Played on Sleeper — ESPN data not authoritative.",
}
```

### Step 2 — Apply the inputs

Don't batch these — show the user each diff before moving on.

**Edit `pipeline/config.py`:**

- Set `LEAGUE_ID`, `FIRST_YEAR`, `CURRENT_YEAR`.
- Optionally update `EXCLUDED_RECORD_YEARS`.
- Update `KEEPER_RULES`.

**Clear `pipeline/overrides.py`:**

This file contains hand-written ID mappings that are specific to Bennett's
league (Jordan Klein → Ben Goldstein co-owner merge, Ben Wolbransky alias
fix, etc.). Strip all of:

- `CO_OWNER_MERGES`
- `ALIAS_MERGES`
- `CO_OWNER_SOURCE_NAMES`
- `DISPLAY_NAME_OVERRIDES`
- `NAME_RENAMES`

…down to empty dicts/sets. The functions and the calling convention stay;
the data tables go to empty. We'll re-populate them in Step 4 once the user
sees their names.

**Delete the stale dataset:**

```bash
rm -rf data/seasons/* data/*.json
```

(Keep the `data/` directory and `data/seasons/` subdirectory.) The pipeline
will rebuild everything from ESPN.

**Rename branding in `web/`:**

Find every occurrence of `"Steak Frites"` and the 2-letter mark `SF` (in
the header/footer logo squares in `web/src/app/layout.tsx`) and replace with

### Step 3 — Run the pipeline

Install Python deps (3.12+):

```bash
pip install -r requirements.txt
```

Run the full pipeline with the user's cookies. **Have the user paste them
into the shell only — never write them into a file in the repo.**

```bash
ESPN_S2='<long-opaque-string>' SWID='{XXXX-XXXX-XXXX-XXXX-XXXX}' \
  python -m pipeline.build
```

What the pipeline does (in order, from `pipeline/build.py`):

1. `pull.py` — Walks every year in `FIRST_YEAR..CURRENT_YEAR`. For each
   season, hits the ESPN API via `espn_api.football.League`, dumps:
   teams, members, draft picks, every weekly matchup, every box-score row.
   Writes `data/seasons/<year>.json`.
2. `postprocess.py` — Reapplies overrides (empty at this point — fine)
   and writes `data/owners.json` and `data/meta.json`.
3. `records.py` — Computes all-time records (blowouts, highs, streaks,
   champions/runners/thirds/sackos, head-to-head, owner-alltime).
   Writes `data/records.json`.
4. `keepers.py` — Reads each season's draft and the previous year's final
   roster, classifies each keeper, writes `data/keepers.json`.
5. `players.py` — Flattens every box-score row into player-week facts,
   computes per-position all-time leaders, season MVPs, "most often on
   winning rosters." Writes `data/players.json`.

**If `pull.py` errors with 401/403:** the cookies are wrong or expired.
Have the user grab a fresh `espn_s2` and `SWID` from a logged-in browser.

**If a specific year errors but others succeed:** ESPN sometimes returns
incomplete data for very old seasons. Note which year and continue — the
site handles a missing season gracefully.

### Step 4 — Name cleanup

Open `data/owners.json` in a text editor (or `cat` the `display_name`
fields) and show the user the list. They will likely spot at least one of:

- A name in lowercase that should be proper-case ("sam jones" → "Sam
  Jones") → add to `DISPLAY_NAME_OVERRIDES` in `pipeline/overrides.py`,
  keyed by the ESPN owner ID (the long `{...}` UUID).
- Two ESPN accounts that are the same person (rejoined under a new login)
  → add to `ALIAS_MERGES`.
- Two ESPN accounts that are co-owners of one team → add to
  `CO_OWNER_MERGES` (and the merged-from display name to
  `CO_OWNER_SOURCE_NAMES`).

After editing, re-run **just** the post-pull stages (no need to re-pull
ESPN, which is slow):

```bash
python -m pipeline.postprocess && \
python -m pipeline.records && \
python -m pipeline.keepers && \
python -m pipeline.players
```

Repeat until the user is happy with how names appear in
`data/owners.json` and `data/records.json`.

### Step 5 — Run the site locally

```bash
cd web
npm install
npm run dev    # http://localhost:3000  (or :5190 if that script is the one shipped)
```

Have the user click through every tab and verify:

- Home reigning-champion block shows their most recent champion + the
  Playoff MVP from their championship roster.
- `/champions` (Trophies) — every year's podium shows up.
- `/playoffs` — playoff leaderboard + finals history.
- `/what-if` — luck index, most snake-bitten seasons.
- `/teams` — owner profiles all link.
- `/keepers` — keeper planner respects their rules. Try selecting an
  illegal combination; the planner should reject it with the right reason.

Run a production build to catch type errors:

```bash
npm run build
```

### Step 6 — Deploy

1. Push the repo to the user's own GitHub.
2. Create a Vercel project pointing at the new repo, with
   **root directory = `web/`**.
3. Build command: default (`next build`). Output: default.
4. No build-time env vars are required — the site reads the committed
   JSON in `data/`.

### Step 7 — Wire up the weekly refresh (optional)

The included `.github/workflows/refresh.yml` runs the pipeline every
Tuesday during the season, commits any data diff to `main`, and lets
Vercel redeploy. To make it work:

1. In the new GitHub repo: **Settings → Secrets and variables → Actions →
   New repository secret**.
2. Add `ESPN_S2` (the cookie value) and `SWID` (with the braces).
3. Push the workflow file. Confirm it appears in the Actions tab.
4. Trigger a manual run from the Actions tab to verify before the first
   scheduled run.

## Reference: where things live

- **Routes**: each tab is a folder under `web/src/app/`. Add a new tab by
  creating `web/src/app/<route>/page.tsx` and adding `{ href, label }` to
  the `NAV` array in `web/src/app/layout.tsx`.
- **Derived stats** (playoff record, all-play, Pythagorean expected wins):
  `web/src/lib/derived.ts`. Pure server-side, reads the JSON data files.
- **Sortable tables**: generic primitive at
  `web/src/components/SortableTable.tsx`. Column definitions must live in
  a `"use client"` wrapper component (functions can't cross the
  server-component → client-component boundary). See
  `PlayoffOwnersTable.tsx` and `WhatIfTables.tsx` for the pattern.
- **Types**: `web/src/lib/types.ts` mirrors the JSON shapes produced by
  the pipeline. If you add a field in Python, add it here too or
  `npm run build` will type-error.

## Things to tell the user up front

- This is a personal weekend project, not a polished product. The styling
  is opinionated (cream + forest green + gold). They are encouraged to
  rip out the color palette and rebrand — Tailwind classes in
  `web/src/app/globals.css` define the base palette.
- The pipeline is **slow on first run** (5–10 minutes for 10 years of
  data, mostly because ESPN throttles historical-season requests). After
  that, only the most recent season needs a refresh.
- Cookies expire when the user logs out of ESPN. If the weekly refresh
  workflow starts failing months from now, that's almost certainly why —
  grab fresh ones and update the GitHub Secrets.
